ğŸ“š Overview: Two Special Workload Types

While Deployments and StatefulSets handle most application workloads, Kubernetes provides specialized controllers for specific use cases:

DaemonSets

Purpose: Run a copy of a pod on every node (or selected nodes)

Use Cases: Logging agents, monitoring agents, network plugins, storage daemons

Jobs

Purpose: Run pods to completion (batch processing)

Use Cases: Data processing, backups, database migrations, report generation

CronJobs

Purpose: Run Jobs on a schedule

Use Cases: Scheduled backups, periodic cleanup, report generation

ğŸ”„ Part 1: DaemonSets

What are DaemonSets?

A DaemonSet ensures that all (or some) nodes run a copy of a pod. As nodes are added to the cluster, pods are automatically added to them. As nodes are removed, those pods are garbage collected.

ğŸ¯ Key Concept

Think of DaemonSets like installing antivirus software on every computer in an office. Just as you want antivirus on every machine, DaemonSets ensure certain pods (like logging or monitoring agents) run on every node in your cluster.

DaemonSet Characteristics

  • One Pod Per Node: Automatically runs on all nodes
  • Automatic Scheduling: New nodes get the pod automatically
  • Node Affinity: Can target specific nodes
  • No Replica Count: Number of pods = number of matching nodes
  • System-Level: Often used for cluster infrastructure

Common Use Cases for DaemonSets

  • Log Collection: Fluentd, Filebeat, Logstash
  • Monitoring: Prometheus Node Exporter, Datadog agent

  • Storage: Ceph, GlusterFS daemons
  • Security: Security scanning agents
  • Performance: Performance monitoring tools

Creating a DaemonSet

Example 1: Simple Logging DaemonSet

# fluentd-daemonset.yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
  namespace: kube-system
  labels:
    app: fluentd
spec:
  selector:
    matchLabels:
      app: fluentd
  template:
    metadata:
      labels:
        app: fluentd
    spec:
      containers:
      - name: fluentd
        image: fluent/fluentd:v1.14
        resources:
          limits:
            memory: 200Mi
          requests:
            cpu: 100m
            memory: 200Mi
        volumeMounts:
        - name: varlog
          mountPath: /var/log
        - name: varlibdockercontainers
          mountPath: /var/lib/docker/containers
          readOnly: true
      
      # Volumes to access node logs
      volumes:
      - name: varlog
        hostPath:
          path: /var/log
      - name: varlibdockercontainers
        hostPath:
          path: /var/lib/docker/containers

Running DaemonSet on Specific Nodes

Method 1: Node Selector

# Label nodes
kubectl label nodes node-1 disktype=ssd
kubectl label nodes node-2 disktype=ssd

# DaemonSet with node selector
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: ssd-monitor
spec:
  selector:
    matchLabels:
      app: ssd-monitor
  template:
    metadata:
      labels:
        app: ssd-monitor
    spec:
      nodeSelector:
        disktype: ssd  # Only runs on nodes with this label
      containers:
      - name: monitor
        image: monitoring-agent:v1

# This DaemonSet only runs on node-1 and node-2

Method 2: Node Affinity

apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: gpu-monitor
spec:
  selector:
    matchLabels:
      app: gpu-monitor
  template:
    metadata:
      labels:
        app: gpu-monitor
    spec:
      affinity:
        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
            - matchExpressions:
              - key: gpu
                operator: In
                values:
                - "true"
      containers:
      - name: monitor
        image: gpu-monitor:v1

Tolerations for Master Nodes

By default, DaemonSets don't run on master nodes (they have taints). To run on masters:

apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-exporter
spec:
  selector:
    matchLabels:
      app: node-exporter
  template:
    metadata:
      labels:
        app: node-exporter
    spec:
      # Tolerate master node taints
      tolerations:
      - key: node-role.kubernetes.io/master
        effect: NoSchedule
      - key: node-role.kubernetes.io/control-plane
        effect: NoSchedule
      
      containers:
      - name: node-exporter
        image: prom/node-exporter:latest
        ports:
        - containerPort: 9100
          hostPort: 9100  # Expose on node's port

# Now runs on ALL nodes including masters

Updating DaemonSets

Update Strategy: RollingUpdate (Default)

apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
spec:
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1  # Update one node at a time
  # ... rest of spec

# Update the image
kubectl set image daemonset/fluentd fluentd=fluent/fluentd:v1.15 -n kube-system

# Watch the rollout
kubectl rollout status daemonset/fluentd -n kube-system

# Rollback if needed
kubectl rollout undo daemonset/fluentd -n kube-system

Update Strategy: OnDelete

apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
spec:
  updateStrategy:
    type: OnDelete  # Manual control
  # ... rest of spec

# Update image
kubectl set image daemonset/fluentd fluentd=fluent/fluentd:v1.15 -n kube-system

# Pods won't update automatically
# Manually delete pods to trigger update
kubectl delete pod fluentd-abc12 -n kube-system
# New pod created with updated image

â±ï¸ Part 2: Jobs

What are Jobs?

A Job creates one or more pods and ensures that a specified number of them successfully terminate. Jobs track successful completions and when the specified number is reached, the Job is complete.

ğŸ¯ Key Concept

Think of Jobs like running a script or batch process. Unlike Deployments that keep pods running forever, Jobs run pods until they complete their task and exit successfully. It's like running a one-time data migration or generating a monthly report.

Job Characteristics

  • Run to Completion: Pods run until they exit successfully
  • Retry on Failure: Automatically retries failed pods
  • Parallel Execution: Can run multiple pods in parallel
  • Completion Tracking: Tracks successful completions
  • Cleanup: Completed pods remain for log inspection

Job vs Deployment

Aspect Deployment Job
Purpose Long-running services Batch/one-time tasks
Lifecycle Runs forever Runs until completion
Restart Always restart Retry on failure only
Success Running = success Exit code 0 = success
Use Case Web servers, APIs Data processing, backups

Creating a Simple Job

# simple-job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: pi-calculation
spec:
  template:
    spec:
      containers:
      - name: pi
        image: perl:5.34
        command: ["perl", "-Mbignum=bpi", "-wle", "print bpi(2000)"]
      restartPolicy: Never  # Never or OnFailure (not Always!)
  
  backoffLimit: 4  # Retry up to 4 times on failure

kubectl apply -f simple-job.yaml

Job Patterns

Pattern 1: Single Job (Default)

Run one pod to completion

apiVersion: batch/v1
kind: Job
metadata:
  name: backup-job
spec:
  completions: 1  # Run 1 pod (default)
  parallelism: 1  # Run 1 at a time (default)
  template:
    spec:
      containers:
      - name: backup
        image: backup-tool:v1
        command: ["./backup.sh"]
      restartPolicy: OnFailure

Pattern 2: Parallel Jobs with Fixed Completion Count

Run multiple pods in parallel until N completions

apiVersion: batch/v1
kind: Job
metadata:
  name: parallel-job
spec:
  completions: 10      # Need 10 successful completions
  parallelism: 3       # Run 3 pods at a time
  template:
    spec:
      containers:
      - name: worker
        image: worker:v1
        command: ["./process-item.sh"]
      restartPolicy: OnFailure

# Execution:
# - Starts 3 pods
# - When one completes, starts another
# - Continues until 10 completions
# - Total pods created: 10 (if all succeed)

Pattern 3: Work Queue (Parallel Processing)

Multiple pods process items from a queue

apiVersion: batch/v1
kind: Job
metadata:
  name: queue-job
spec:
  parallelism: 5       # Run 5 workers
  # No completions specified - workers decide when done
  template:
    spec:
      containers:
      - name: worker
        image: queue-worker:v1
        env:
        - name: QUEUE_URL
          value: "redis://redis:6379"
        command:
        - /bin/sh
        - -c
        - |
          while true; do
            item=$(redis-cli lpop work-queue)
            if [ -z "$item" ]; then
              echo "Queue empty, exiting"
              exit 0
            fi
            echo "Processing: $item"
            ./process.sh "$item"
          done
      restartPolicy: OnFailure

# Workers pull items from queue until empty

Job Configuration Options

apiVersion: batch/v1
kind: Job
metadata:
  name: configured-job
spec:
  # Completion settings
  completions: 5              # Number of successful completions needed
  parallelism: 2              # Number of pods to run in parallel
  
  # Failure handling
  backoffLimit: 6             # Max retries before marking as failed
  activeDeadlineSeconds: 600  # Max time job can run (10 minutes)
  
  # Pod lifecycle
  ttlSecondsAfterFinished: 100  # Delete job 100s after completion
  
  template:
    spec:
      containers:
      - name: worker
        image: worker:v1
      restartPolicy: OnFailure  # Never or OnFailure only
ğŸ’¡ Key Settings:
  • backoffLimit: How many times to retry failed pods
  • activeDeadlineSeconds: Timeout for entire job
  • ttlSecondsAfterFinished: Auto-cleanup completed jobs
  • restartPolicy: Must be Never or OnFailure (not Always)

Real-World Example: Database Backup Job

# postgres-backup-job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: postgres-backup
  labels:
    app: postgres-backup
spec:
  backoffLimit: 3
  activeDeadlineSeconds: 3600  # 1 hour timeout
  template:
    metadata:
      labels:
        app: postgres-backup
    spec:
      containers:
      - name: backup
        image: postgres:14
        command:
        - /bin/sh
        - -c
        - |
          echo "Starting backup..."
          TIMESTAMP=$(date +%Y%m%d-%H%M%S)
          BACKUP_FILE="/backup/postgres-backup-${TIMESTAMP}.sql"
          
          pg_dump -h postgres -U postgres -d mydb > ${BACKUP_FILE}
          
          if [ $? -eq 0 ]; then
            echo "Backup successful: ${BACKUP_FILE}"
            # Upload to S3 or other storage
            # aws s3 cp ${BACKUP_FILE} s3://my-backups/
            exit 0
          else
            echo "Backup failed!"
            exit 1
          fi
        env:
        - name: PGPASSWORD
          valueFrom:
            secretKeyRef:
              name: postgres-secret
              key: password
        volumeMounts:
        - name: backup-storage
          mountPath: /backup
      
      volumes:
      - name: backup-storage
        persistentVolumeClaim:
          claimName: backup-pvc
      
      restartPolicy: OnFailure

kubectl apply -f postgres-backup-job.yaml

# Monitor job
kubectl get job postgres-backup -w

# View logs
kubectl logs -l app=postgres-backup

# Check backup file
kubectl exec -it postgres-backup-abc12 -- ls -lh /backup

â° Part 3: CronJobs

What are CronJobs?

CronJobs create Jobs on a schedule, similar to cron in Linux. They're perfect for recurring tasks like backups, reports, or cleanup.

ğŸ¯ Key Concept

Think of CronJobs like scheduled tasks on your computer. Just as you might schedule a weekly virus scan or daily backup, CronJobs run tasks on a schedule in Kubernetes.

Cron Schedule Format

â”Œâ”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€ minute (0 - 59)
â”‚ â”Œâ”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€ hour (0 - 23)
â”‚ â”‚ â”Œâ”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€ day of month (1 - 31)
â”‚ â”‚ â”‚ â”Œâ”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€ month (1 - 12)
â”‚ â”‚ â”‚ â”‚ â”Œâ”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€ day of week (0 - 6) (Sunday=0)
â”‚ â”‚ â”‚ â”‚ â”‚
* * * * *

Examples:
"0 0 * * *"      # Daily at midnight
"0 */6 * * *"    # Every 6 hours
"0 9 * * 1"      # Every Monday at 9 AM
"*/15 * * * *"   # Every 15 minutes
"0 0 1 * *"      # First day of every month at midnight
"0 2 * * 0"      # Every Sunday at 2 AM

Creating a CronJob

Example 1: Daily Backup

# daily-backup-cronjob.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: daily-backup
spec:
  schedule: "0 2 * * *"  # Every day at 2 AM
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: backup-tool:v1
            command:
            - /bin/sh
            - -c
            - |
              echo "Running backup at $(date)"
              ./backup.sh
              echo "Backup completed"
          restartPolicy: OnFailure
  
  # CronJob specific settings
  successfulJobsHistoryLimit: 3  # Keep last 3 successful jobs
  failedJobsHistoryLimit: 1      # Keep last 1 failed job
  concurrencyPolicy: Forbid       # Don't run if previous job still running

kubectl apply -f daily-backup-cronjob.yaml

# View CronJob
kubectl get cronjob

# Output:
# NAME           SCHEDULE      SUSPEND   ACTIVE   LAST SCHEDULE   AGE
# daily-backup   0 2 * * *     False     0                  10s

# Manually trigger a job (for testing)
kubectl create job --from=cronjob/daily-backup manual-backup-1

# View jobs created by CronJob
kubectl get jobs -l job-name=daily-backup

CronJob Concurrency Policies

  • Allow (default): Multiple jobs can run concurrently
  • Forbid: Skip new job if previous is still running
  • Replace: Cancel running job and start new one
apiVersion: batch/v1
kind: CronJob
metadata:
  name: cleanup-job
spec:
  schedule: "*/5 * * * *"  # Every 5 minutes
  concurrencyPolicy: Forbid  # Don't overlap
  startingDeadlineSeconds: 300  # If missed, don't start after 5 min
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 1
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: cleanup
            image: busybox
            command:
            - /bin/sh
            - -c
            - |
              echo "Cleaning up old files..."
              find /data -type f -mtime +7 -delete
              echo "Cleanup complete"
          restartPolicy: OnFailure

# db-cleanup-cronjob.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: db-cleanup
  namespace: production
spec:
  schedule: "0 3 * * 0"  # Every Sunday at 3 AM
  concurrencyPolicy: Forbid
  successfulJobsHistoryLimit: 5
  failedJobsHistoryLimit: 3
  jobTemplate:
    spec:
      backoffLimit: 2
      activeDeadlineSeconds: 1800  # 30 minutes
      template:
        spec:
          containers:
          - name: cleanup
            image: postgres:14
            command:
            - /bin/sh
            - -c
            - |
              echo "Starting database cleanup..."
              
              # Delete old logs
              psql -h postgres -U postgres -d mydb -c "
                DELETE FROM logs WHERE created_at < NOW() - INTERVAL '90 days';
              "
              
              # Delete old sessions
              psql -h postgres -U postgres -d mydb -c "
                DELETE FROM sessions WHERE last_activity < NOW() - INTERVAL '30 days';
              "
              
              # Vacuum database
              psql -h postgres -U postgres -d mydb -c "VACUUM ANALYZE;"
              
              echo "Cleanup completed successfully"
            env:
            - name: PGPASSWORD
              valueFrom:
                secretKeyRef:
                  name: postgres-secret
                  key: password
          restartPolicy: OnFailure

kubectl apply -f db-cleanup-cronjob.yaml

ğŸ”§ Init Containers

Init containers run before app containers and are useful for setup tasks.

What are Init Containers?

  • Run Before App: Execute before main containers start
  • Run Sequentially: One at a time, in order
  • Must Complete: App containers don't start until all init containers succeed
  • Use Cases: Wait for dependencies, setup, data loading

Init Container Examples

Example 1: Wait for Service

apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  initContainers:
  - name: wait-for-db
    image: busybox:1.28
    command:
    - sh
    - -c
    - |
      echo "Waiting for database..."
      until nslookup postgres.default.svc.cluster.local; do
        echo "Database not ready, waiting..."
        sleep 2
      done
      echo "Database is ready!"
  
  containers:
  - name: app
    image: myapp:v1
    # This starts only after init container completes

Example 2: Download Configuration

apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  initContainers:
  - name: download-config
    image: busybox:1.28
    command:
    - sh
    - -c
    - |
      echo "Downloading configuration..."
      wget -O /config/app.conf https://config-server/app.conf
      echo "Configuration downloaded"
    volumeMounts:
    - name: config
      mountPath: /config
  
  containers:
  - name: app
    image: myapp:v1
    volumeMounts:
    - name: config
      mountPath: /etc/app
  
  volumes:
  - name: config
    emptyDir: {}
