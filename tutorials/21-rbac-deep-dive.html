ğŸ¤” What is RBAC?

Role-Based Access Control (RBAC) is a method of regulating access to resources based on the roles of individual users within an organization. In Kubernetes, RBAC determines who can perform what actions on which resources.

RBAC Core Concepts

  1. Subject: User, Group, or ServiceAccount that wants to access resources
  2. Resource: Kubernetes objects (pods, services, deployments, etc.)
  3. Verb: Action to perform (get, list, create, update, delete, etc.)
  4. Role/ClusterRole: Set of permissions (rules)
  5. RoleBinding/ClusterRoleBinding: Grants permissions to subjects

ğŸ“‹ Role vs ClusterRole

Key Differences

Aspect Role ClusterRole
Scope Namespace-scoped Cluster-wide
Resources Namespaced resources only All resources (including cluster-scoped)
Use Case Team/project permissions Admin, cross-namespace, cluster resources
Examples Developer in 'dev' namespace Cluster admin, node viewer

Creating a Role

# role-developer.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: developer
  namespace: dev
rules:
# Allow all operations on pods
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]

# Allow reading pod logs
- apiGroups: [""]
  resources: ["pods/log"]
  verbs: ["get", "list"]

# Allow exec into pods
- apiGroups: [""]
  resources: ["pods/exec"]
  verbs: ["create"]

# Allow all operations on deployments
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]

# Allow reading services
- apiGroups: [""]
  resources: ["services"]
  verbs: ["get", "list", "watch"]

# Allow reading configmaps and secrets
- apiGroups: [""]
  resources: ["configmaps", "secrets"]
  verbs: ["get", "list", "watch"]

# Apply the role
kubectl apply -f role-developer.yaml

# View the role
kubectl describe role developer -n dev

Creating a ClusterRole

# clusterrole-node-reader.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: node-reader
rules:
# Allow reading nodes (cluster-scoped resource)
- apiGroups: [""]
  resources: ["nodes"]
  verbs: ["get", "list", "watch"]

# Allow reading node metrics
- apiGroups: [""]
  resources: ["nodes/status"]
  verbs: ["get"]

# Allow reading persistent volumes (cluster-scoped)
- apiGroups: [""]
  resources: ["persistentvolumes"]
  verbs: ["get", "list", "watch"]

# Apply the cluster role
kubectl apply -f clusterrole-node-reader.yaml

# View the cluster role
kubectl describe clusterrole node-reader

ğŸ”— RoleBinding vs ClusterRoleBinding

Bindings grant the permissions defined in a Role or ClusterRole to subjects (users, groups, or service accounts).

RoleBinding Examples

# rolebinding-developer.yaml
# Bind Role to a User
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: developer-binding
  namespace: dev
subjects:
- kind: User
  name: john@example.com
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer
  apiGroup: rbac.authorization.k8s.io

---
# Bind Role to a Group
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: developers-group-binding
  namespace: dev
subjects:
- kind: Group
  name: developers
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer
  apiGroup: rbac.authorization.k8s.io

---
# Bind Role to a ServiceAccount
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-sa-binding
  namespace: dev
subjects:
- kind: ServiceAccount
  name: myapp-sa
  namespace: dev
roleRef:
  kind: Role
  name: developer
  apiGroup: rbac.authorization.k8s.io

# Apply bindings
kubectl apply -f rolebinding-developer.yaml

# View role bindings
kubectl get rolebindings -n dev
kubectl describe rolebinding developer-binding -n dev

ClusterRoleBinding Examples

# clusterrolebinding-node-reader.yaml
# Grant cluster-wide permissions
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: node-reader-binding
subjects:
- kind: User
  name: monitoring@example.com
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: node-reader
  apiGroup: rbac.authorization.k8s.io

---
# Grant ClusterRole to ServiceAccount
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: prometheus-binding
subjects:
- kind: ServiceAccount
  name: prometheus
  namespace: monitoring
roleRef:
  kind: ClusterRole
  name: cluster-monitoring
  apiGroup: rbac.authorization.k8s.io

# Apply cluster role bindings
kubectl apply -f clusterrolebinding-node-reader.yaml

# View cluster role bindings
kubectl get clusterrolebindings
kubectl describe clusterrolebinding node-reader-binding

âš ï¸ Important: RoleBinding with ClusterRole

You can bind a ClusterRole using a RoleBinding to grant permissions only within that namespace. This is useful for reusing common ClusterRoles across namespaces.

# Use ClusterRole 'view' but limit to 'dev' namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: dev
subjects:
- kind: User
  name: jane@example.com
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole  # Note: ClusterRole, not Role
  name: view         # Built-in ClusterRole
  apiGroup: rbac.authorization.k8s.io

ğŸ‘¤ Service Accounts

Service accounts provide an identity for processes running in pods.

Creating and Using Service Accounts

# Create service account
kubectl create serviceaccount myapp-sa -n production

# Or using YAML
apiVersion: v1
kind: ServiceAccount
metadata:
  name: myapp-sa
  namespace: production

# Create Role for the service account
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: myapp-role
  namespace: production
rules:
- apiGroups: [""]
  resources: ["configmaps", "secrets"]
  verbs: ["get", "list"]
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]

# Bind Role to ServiceAccount
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: myapp-binding
  namespace: production
subjects:
- kind: ServiceAccount
  name: myapp-sa
  namespace: production
roleRef:
  kind: Role
  name: myapp-role
  apiGroup: rbac.authorization.k8s.io

# Use ServiceAccount in Pod
apiVersion: v1
kind: Pod
metadata:
  name: myapp
  namespace: production
spec:
  serviceAccountName: myapp-sa
  containers:
  - name: myapp
    image: myapp:v1.0.0

# View service account
kubectl get serviceaccount myapp-sa -n production -o yaml

# Get service account token (for external access)
kubectl create token myapp-sa -n production --duration=24h

Service Account Token Projection

# Modern approach: Projected service account tokens
apiVersion: v1
kind: Pod
metadata:
  name: myapp
  namespace: production
spec:
  serviceAccountName: myapp-sa
  containers:
  - name: myapp
    image: myapp:v1.0.0
    volumeMounts:
    - name: token
      mountPath: /var/run/secrets/tokens
      readOnly: true
  volumes:
  - name: token
    projected:
      sources:
      - serviceAccountToken:
          path: token
          expirationSeconds: 3600  # 1 hour
          audience: api

# Access token from within pod
cat /var/run/secrets/tokens/token

# Use token to access Kubernetes API
TOKEN=$(cat /var/run/secrets/tokens/token)
curl -H "Authorization: Bearer $TOKEN" \
  --cacert /var/run/secrets/kubernetes.io/serviceaccount/ca.crt \
  https://kubernetes.default.svc/api/v1/namespaces/production/pods

ğŸ¯ Common RBAC Patterns

Pattern 1: Read-Only Access

# ClusterRole for read-only access to all resources
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: read-only
rules:
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["get", "list", "watch"]

# Bind to user
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: read-only-binding
subjects:
- kind: User
  name: auditor@example.com
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: read-only
  apiGroup: rbac.authorization.k8s.io

# Or use built-in 'view' ClusterRole
kubectl create clusterrolebinding view-all \
  --clusterrole=view \
  --user=viewer@example.com

Pattern 2: Namespace Admin

# Full admin access to a specific namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: dev-admin
  namespace: dev
subjects:
- kind: User
  name: dev-lead@example.com
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: admin  # Built-in admin ClusterRole
  apiGroup: rbac.authorization.k8s.io

# Or create custom namespace admin role
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: namespace-admin
  namespace: dev
rules:
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["*"]

# Bind custom role
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: namespace-admin-binding
  namespace: dev
subjects:
- kind: User
  name: dev-lead@example.com
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: namespace-admin
  apiGroup: rbac.authorization.k8s.io

Pattern 3: CI/CD Pipeline Access

# ServiceAccount for CI/CD
apiVersion: v1
kind: ServiceAccount
metadata:
  name: cicd-deployer
  namespace: production

---
# Role for deployment
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: deployer
  namespace: production
rules:
# Manage deployments
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]

# Manage services
- apiGroups: [""]
  resources: ["services"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]

# Manage configmaps
- apiGroups: [""]
  resources: ["configmaps"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]

# Read secrets (but not create/delete)
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list"]

# View pods and logs
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]

---
# Bind role to service account
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: cicd-deployer-binding
  namespace: production
subjects:
- kind: ServiceAccount
  name: cicd-deployer
  namespace: production
roleRef:
  kind: Role
  name: deployer
  apiGroup: rbac.authorization.k8s.io

# Get token for CI/CD
kubectl create token cicd-deployer -n production --duration=8760h  # 1 year

ğŸ” Testing and Auditing Permissions

Using kubectl auth can-i

# Check if current user can create pods
kubectl auth can-i create pods -n production

# Check if current user can delete deployments
kubectl auth can-i delete deployments -n production

# Check all permissions for current user
kubectl auth can-i --list -n production

# Check permissions for specific user
kubectl auth can-i create pods -n production --as john@example.com

# Check permissions for service account
kubectl auth can-i list secrets -n production \
  --as system:serviceaccount:production:myapp-sa

# Check permissions for group
kubectl auth can-i delete pods -n production --as-group=developers

# Check all verbs for a resource
kubectl auth can-i '*' pods -n production

# Output examples:
# yes - permission granted
# no - permission denied
# no - no RBAC policy matched

Auditing RBAC Policies

# List all roles in a namespace
kubectl get roles -n production

# List all cluster roles
kubectl get clusterroles

# List all role bindings in a namespace
kubectl get rolebindings -n production

# List all cluster role bindings
kubectl get clusterrolebindings

# Find all bindings for a specific user
kubectl get rolebindings,clusterrolebindings -A -o json | \
  jq -r '.items[] | select(.subjects[]?.name=="john@example.com") | 
    "\(.metadata.namespace // "cluster"):\(.metadata.name)"'

# Find all bindings for a service account
kubectl get rolebindings,clusterrolebindings -A -o json | \
  jq -r '.items[] | select(.subjects[]?.name=="myapp-sa") | 
    "\(.metadata.namespace // "cluster"):\(.metadata.name)"'

# Find who has cluster-admin access
kubectl get clusterrolebindings -o json | \
  jq -r '.items[] | select(.roleRef.name=="cluster-admin") | 
    .subjects[]?.name'

# Describe specific role
kubectl describe role developer -n dev

# Describe specific cluster role
kubectl describe clusterrole admin

# Export all RBAC policies
kubectl get roles,rolebindings,clusterroles,clusterrolebindings -A -o yaml > rbac-backup.yaml

ğŸ”§ Aggregated ClusterRoles

Aggregated ClusterRoles combine multiple ClusterRoles using label selectors.

# Parent aggregated ClusterRole
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: monitoring-aggregate
aggregationRule:
  clusterRoleSelectors:
  - matchLabels:
      rbac.example.com/aggregate-to-monitoring: "true"
rules: []  # Rules are automatically filled

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: monitoring-pods
  labels:
    rbac.example.com/aggregate-to-monitoring: "true"
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]

---
# Child ClusterRole 2: Node monitoring
apiVersion: rbac.authorization.k8s.io/v1

  name: monitoring-nodes

    rbac.example.com/aggregate-to-monitoring: "true"
rules:
- apiGroups: [""]
  resources: ["nodes", "nodes/metrics"]
  verbs: ["get", "list", "watch"]

---
# Child ClusterRole 3: Service monitoring
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: monitoring-services

    rbac.example.com/aggregate-to-monitoring: "true"

- apiGroups: [""]

# The monitoring-aggregate ClusterRole now includes all rules
# from the child ClusterRoles

# Bind aggregated role
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: monitoring-binding
subjects:
- kind: ServiceAccount
  name: monitoring-sa
  namespace: monitoring
roleRef:
  kind: ClusterRole
  name: monitoring-aggregate
  apiGroup: rbac.authorization.k8s.io

ğŸ›¡ï¸ Least Privilege Principle

Always grant the minimum permissions necessary for a task.

Best Practices

  1. Start with no permissions: Grant only what's needed
  2. Use specific resources: Avoid wildcards (*) when possible
  3. Limit verbs: Don't grant 'delete' unless necessary
  4. Namespace scope: Use Roles instead of ClusterRoles when possible
  5. Regular audits: Review and remove unused permissions
  6. Separate concerns: Different roles for different tasks
# âŒ BAD: Too permissive
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: bad-role
rules:
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["*"]

# âœ… GOOD: Specific permissions
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: good-role
  namespace: production
rules:
# Only what's needed for the application
- apiGroups: [""]
  resources: ["configmaps"]
  resourceNames: ["app-config"]  # Specific resource
  verbs: ["get"]

- apiGroups: [""]
  resources: ["secrets"]
  resourceNames: ["app-secret"]
  verbs: ["get"]

- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list"]  # No create/delete

# âŒ BAD: Wildcard verbs
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["*"]

# âœ… GOOD: Specific verbs
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "watch", "update", "patch"]  # No delete

# âŒ BAD: Cluster-wide when namespace would work

  name: bad-binding

- kind: User
  name: developer@example.com
roleRef:
  kind: ClusterRole
  name: edit

ğŸ” Troubleshooting RBAC Issues

Common Issues and Solutions

Issue 1: "Forbidden" Error

# Error message:
# Error from server (Forbidden): pods is forbidden: 
# User "john@example.com" cannot list resource "pods" in API group "" in the namespace "production"

# Diagnosis:
# 1. Check if user has any permissions
kubectl auth can-i list pods -n production --as john@example.com

# 2. Check existing role bindings
kubectl get rolebindings -n production -o wide
kubectl get clusterrolebindings -o wide

# 3. Check what permissions user has
kubectl auth can-i --list -n production --as john@example.com

# Solution:
# Create appropriate role and binding
kubectl create role pod-reader --verb=get,list,watch --resource=pods -n production
kubectl create rolebinding john-pod-reader \
  --role=pod-reader \
  --user=john@example.com \
  -n production

Issue 2: ServiceAccount Not Working

# Pod can't access Kubernetes API

# Diagnosis:
# 1. Check if ServiceAccount exists
kubectl get serviceaccount myapp-sa -n production

# 2. Check if pod is using the ServiceAccount
kubectl get pod myapp -n production -o jsonpath='{.spec.serviceAccountName}'

# 3. Check if ServiceAccount has permissions
kubectl auth can-i list pods -n production \
  --as system:serviceaccount:production:myapp-sa

# 4. Check role bindings
kubectl get rolebindings,clusterrolebindings -A -o json | \
  jq -r '.items[] | select(.subjects[]?.name=="myapp-sa")'

# Solution:
# Ensure ServiceAccount is bound to a role
kubectl create rolebinding myapp-sa-binding \
  --role=myapp-role \
  --serviceaccount=production:myapp-sa \
  -n production

Issue 3: Permissions Work in One Namespace But Not Another

# User can access dev but not production

# Diagnosis:
# 1. Check RoleBindings in both namespaces
kubectl get rolebindings -n dev
kubectl get rolebindings -n production

# 2. Check if using ClusterRoleBinding (applies everywhere)
kubectl get clusterrolebindings

# 3. Verify permissions in each namespace
kubectl auth can-i list pods -n dev --as john@example.com
kubectl auth can-i list pods -n production --as john@example.com

# Solution:
# RoleBindings are namespace-scoped, create one for production
kubectl create rolebinding john-production \
  --clusterrole=view \
  --user=john@example.com \
  -n production

ğŸ“Š Built-in ClusterRoles

Kubernetes provides several built-in ClusterRoles for common use cases.

ClusterRole Description Use Case
cluster-admin Full cluster access Cluster administrators
admin Full namespace access Namespace administrators
edit Read/write in namespace Developers
view Read-only in namespace Viewers, auditors
system:* System components Internal use only 
# View built-in ClusterRoles
kubectl get clusterroles

# Describe a built-in ClusterRole
kubectl describe clusterrole view
kubectl describe clusterrole edit
kubectl describe clusterrole admin
kubectl describe clusterrole cluster-admin

# Use built-in ClusterRoles
# Grant cluster-admin to user (full access)
kubectl create clusterrolebinding admin-user \
  --clusterrole=cluster-admin \
  --user=admin@example.com

# Grant namespace admin
kubectl create rolebinding namespace-admin \
  --clusterrole=admin \
  --user=lead@example.com \
  -n production

# Grant edit access (can modify resources)
kubectl create rolebinding developer \
  --clusterrole=edit \
  --user=dev@example.com \
  -n dev

# Grant view access (read-only)
kubectl create rolebinding viewer \
  --clusterrole=view \
  --user=viewer@example.com \
  -n production
